# Lessons

このプロジェクトで受けた修正・失敗から学んだパターンを記録する。
セッション開始時にレビューし、同じミスを繰り返さない。

## 2026-07-20: HTMLの`<script src>`にクエリを付ける編集で閉じタグを落とした

**症状**: GitHub Pagesで「動かない」。app.jsの関数が全て`undefined`、予想もボタンも何も描画されず。

**原因**: `<script src="app.js"></script>` にバージョンクエリを付ける編集で、`</script>` 閉じタグを削除してしまった（`<script src="app.js?v=...">` の状態）。閉じタグがないと後続の`</body>`等がスクリプト内容として飲み込まれ、外部スクリプトが実行されない。

**誤診も記録**: 当初「キャッシュ破損」と誤診した。`transferSize:0`（キャッシュ読込）に引きずられたが、真因はHTMLの構文破壊だった。動的に`createElement('script')`で注入すると動いたため（=コード自体は正常）、切り分けで確定できた。

**再発防止（How to apply）**:
- HTMLの`<script>`/`<link>`等のタグを編集したら、必ず開閉タグの対応を確認する（例: `grep -c '<script' / '</script>'` で一致確認）。
- 「動かない」報告では、まずブラウザで関数/DOMの実体を確認する。関数が全て`undefined`＝スクリプト未実行＝**ロード失敗かHTML構文破壊**を最優先で疑う（キャッシュより先に）。
- キャッシュ説に飛びつく前に、`curl`でライブHTMLの該当タグを直接目視する。

**関連**: GitHub Pagesは`pages-build-deployment`ワークフローでデプロイ。GitHub障害(503)時はデプロイが失敗し得るので、`gh run list`でrun結果(success/failure)を確認し、失敗なら`gh run rerun`で再実行する。反映は`curl -sI`の`last-modified`と`age`で判断（HTMLのCDN TTLは約600秒）。

## 2026-07-20: 「決定的」と書いてあったLSTMが実は実行ごとにぶれていた

**症状**: 同一データ・同一条件でワークフローを2回実行したら `analysis_numbers4.json` に差分。
確率がぶれ、期間400/ml_heavy の予想が `2517` → `2518` に反転した。
`rf_scores`/`lstm_scores` を突き合わせた結果、**RFは全期間で完全一致・LSTMだけが全期間でぶれる**と確定。

**外した仮説を全部記録する**（ここが本題）。CI上で実測して以下をすべて否定した:

| 仮説 | 実測結果 |
|---|---|
| CPUマルチスレッドの加算順序 | 否定。同一プロセス／別プロセス／`OMP=MKL=1`固定すべて一致 |
| CPUマイクロアーキ差 | 否定。EPYC 7763 と 9V74 で出力ハッシュが完全一致 |
| RF(`n_jobs=-1`)が先に走る副作用 | 否定。本番と同じRF→LSTM順序でも一致 |
| ライブラリ版の差 | 否定。両実行とも torch 2.13.0+cpu / numpy 2.5.1 / sklearn 1.9.0 |
| 入力データが違った | 否定。draws のハッシュが一致 |

**原因は未特定**。孤立して呼ぶ限りLSTMは別マシンでも再現するのに、
全期間(7030件)や曜日別を含む多数の期間を1プロセスで連続実行する本番パイプラインでは崩れる。

**やらかし**: 最初の診断（スレッドの加算順序）を検証せずに `torch.set_num_threads(1)` と
丸め6桁→4桁で「修正」してコミットした。実測したら**直っておらず、N3は逆に悪化**した
（修正前は一致していたのに差分が出るようになった）。この2つは撤回した。
さらに診断用ワークフローで `run: |` ブロック内のPythonをインデントせず、
YAMLブロックスカラーを途中で終わらせて壊した（GitHubが`workflow_dispatch`を認識せず422）。
**これは lessons.md 冒頭のHTML閉じタグの件と同じ「構造を持つテキストを壊した」パターン**。

**再発防止（How to apply）**:
- 「決定的」と主張するなら**同一条件で2回流して差分ゼロを実測する**。seedを固定した＝決定的、ではない。
- **修正をコミットする前に、その修正が効いたことを実測する**。原因の推定と修正の検証は別物。
  今回は「実測してから主張する」方針にしていたおかげで、直ったつもりの誤修正を出さずに済んだ。
- 原因究明が3つ空振りしたら、原因を追うのをやめて**要件を別の方法で満たせないか**を考える。
  今回の要件は「同じ回号なら予想が変わらない」であって「bit再現性」ではなかった。
  → `analyze_numbers.py` に「データが変わらなければ再計算しない」を入れて解決した。
- YAMLの `run: |` に複数行スクリプトを書くときは**中身を必ずインデント**する。
  長いスクリプトは別ファイルに出す。push前に `yaml.safe_load` で構文確認する。

**関連**: 予想の種は実行日ではなく予想対象の回号（`_seed_key`）。日付依存だと手入力で夜に出した
予想が翌朝の自動更新で別番号に変わってしまうため。ただし再現性が無いので、
種だけでは保証にならず、再計算スキップが実質の担保になっている。
